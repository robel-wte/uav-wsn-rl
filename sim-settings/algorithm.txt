
# --- UAV-WSN Routing Protocol Algorithms (Pseudocode, Publication Format) ---

Algorithm 1: Cluster Formation
Input: Node set N, CH probability P, current round r, residual energy E_i, average residual energy E_avg
Output: Cluster Head set CH, Cluster membership assignments
for each node i in N do
    if i is alive then
        if i not selected as CH in last 1/P rounds then
            T_i ← (P × (E_i / E_avg)) / (1 - P × (r mod (1/P)))
        else
            T_i ← 0
        end if
        x ← UniformRandom(0,1)
        if x < T_i then
            i.status ← CH
            Broadcast(CH_ADV, i)
        else
            WaitFor(CH_ADV)
            ch* ← SelectNearestCH()
            Send(JOIN_REQ, ch*)
        end if
    end if
end for
for each CH in CH do
    Collect(JOIN_REQs)
    FinalizeCluster()
end for

Algorithm 2: Steady-State Data Collection
Input: Cluster assignments, TDMA schedule
Output: Aggregated data at CHs
for each round do
    for each node i not CH do
        WaitFor(TDMA_Slot)
        data ← Sense()
        Send(data, CH)
        SleepUntilNextRound()
    end for
    for each CH do
        ReceiveDataFromMembers()
        agg_data ← AggregateData()
        Buffer(agg_data)
    end for
end for

Algorithm 3: UAV Contact and Data Offloading
Input: UAV trajectory, CH buffer states
Output: Data transferred to UAV
while UAV in operation do
    MoveAlongTrajectory()
    if InCHRange() then
        Broadcast(UAV_BEACON)
        for each CH in range do
            SendData(CH.Buffer, UAV)
            WaitFor(ACK)
            if ACK then
                CH.Buffer ← RemoveSentData()
            end if
        end for
    end if
end while

Algorithm 4: Data Delivery to Base Station
Input: UAV data buffer, BS connection
Output: Data delivered to BS
while UAV has data do
    if ConnectedToBS() then
        Send(UAV.DataBuffer, BS)
        ConfirmDelivery()
    end if
end while

4. Simulation Setup and Methodology
To evaluate the performance of the proposed UAV-assisted routing protocol, a comprehensive simulation environment is developed using OMNeT++ (v6.0 or higher), leveraging the INET Framework and its power consumption modules. This section details the environmental parameters, the energy model, and the benchmark scenarios designed to validate the protocol's effectiveness.
4.1 Simulation Environment and Network Model
The simulation is hosted on an HP Generation 6 Standalone Server, providing a stable and high-performance environment for long-duration discrete-event simulations. The network is modeled as a set of stationary sensor nodes and a single mobile sink (UAV).
4.1.1 Network Configuration
The simulation area is defined as a $100m \times 100m$ square field. Within this area, $N=100$ sensor nodes are deployed using a Uniform Random Distribution. This ensures that the protocol is tested against unpredictable topology densities, providing a more robust evaluation than fixed-grid deployments.
4.1.2 Entity Specifications
•	Sensor Nodes: Each node is initialized with a limited energy budget of $0.5J$. This low energy constraint is specifically chosen to accelerate the arrival of the Stability Period (FND), allowing for multiple lifecycle evaluations within feasible simulation timeframes.
•	Base Station (BS): The BS is located far outside the sensing field ($>200m$ from the boundary) to represent remote monitoring scenarios. The BS has unlimited energy, acting as the final data sink.
•	UAV (Mobile Sink): A single UAV acts as an aerial data collector. Consistent with industrial standards, the UAV flies at a fixed altitude of $H=20m$ to maintain a stable Line-of-Sight (LoS) while avoiding ground obstacles. Its speed varies between $15-20m/s$ to test the link stability ($P_{contact}$) metrics.
4.2 Energy and Communication Model
We adopt the First-Order Radio Model for energy dissipation. This model is widely accepted in benchmark routing research for its clarity in separating electronic circuitry costs from RF amplification costs.
Parameter	Value	Justification
Circuitry Energy ($E_{elec}$)	$50 nJ/bit$	Standard value for low-power sensor transceivers.
Amplifier ($\epsilon_{fs}$)	$10 pJ/bit/m^2$	Free-space model (used for short distances, $d < d_0$).
Control Packet Size	$200 bits$	Accounts for typical header + CH advertising payload.
Initial Energy ($E_0$)	$0.5 J$	Validates the efficiency of the "Stability Period."
CH Probability ($p$)	$0.2$	Optimized starting point for a 100-node network.
4.3 Benchmark Scenarios and Justification
As a benchmark paper, the simulation must be run across distinct scenarios to demonstrate how the protocol scales and adapts.
Scenario	Critical Variable	Justification
1. Scalability Test	Node Density ($50$ to $250$)	Evaluates the Control Packet Overhead ($CO$). As density increases, the efficiency of "Breadcrumb" discovery must outweigh systematic scanning.
2. Mobility Impact	UAV Velocity ($10$ to $30m/s$)	Directly impacts the Contact Period ($P_{contact}$). Tests the UAV's ability to switch to HOVER mode when $T_{cont} < T_{req}$.
3. Energy Sensitivity	Initial Energy ($0.1J$ to $1.0J$)	Measures the protocol’s impact on FND and LND. High-end papers require proof that the protocol is not "energy-greedy."
4. Payload Variability	Data Size ($2k$ to $10k$ bits)	Tests the Throughput and Packet Delivery Ratio (PDR) limits during high-traffic bursts.

5. The Simulation Process
The entire process follows a four-stage execution cycle, ensuring high data fidelity and reproducibility.
Step 1: Topology Generation and Initialization
At $T=0$, the OMNeT++ NED files generate the random deployment. Each node executes Algorithm 1, calculating its CH threshold based on the variable probability $p=0.2$ and its current residual energy.
•	Discussion: This step is critical for setting the "ground truth" of the network health.
Step 2: Discovery and Clustering (Setup Phase)
The UAV begins its raster scan. During this phase, we log all Control Packet Overhead. Nodes exchange ADV and JOIN messages.
•	Discussion: Measuring $CO$ here is essential to prove that the overhead does not consume the entire $0.5J$ budget before data can be sent.
Step 3: Steady-State and Data Collection
Nodes transmit data to CHs; the UAV identifies CHs via Algorithm 2 and executes Algorithm 3 for transfer.
•	Discussion: During each transfer, we record the Timestamp of Generation vs. Timestamp of Receipt at the BS. This raw data is later used to generate the Cumulative Distribution Function (CDF) of end-to-end delay.
Step 4: Metric Logging and Termination
The simulation runs until LND (all nodes die) or the battery threshold is reached.
•	Discussion: After termination, the OMNeT++ .sca and .vec files are processed. We specifically look for the Stability Period (FND) to assess if our energy-aware selection successfully balanced the load across the $100m \times 100m$ area.
