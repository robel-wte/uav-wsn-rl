================================================================================
UAV-WSN ROUTING PROTOCOL ANALYSIS REPORT
================================================================================

1. LEACH CLUSTERING LOGIC ISSUES
--------------------------------------------------------------------------------
* Threshold Reset Logic: 
  Standard LEACH requires a node to be ineligible for CH status for 1/P rounds. 
  Ensure 'calculateThreshold()' properly implements: 
  T(n) = P / (1 - P * (r mod 1/P)). 
  In the current code, verify that 'roundNum' is synchronized globally; 
  if nodes update 'roundNum' independently based on internal timers, 
  clustering will desynchronize, leading to multiple rounds with 0 CHs.

* Energy-Awareness Gap: 
  The base LEACH logic (as seen in SensorNode.cc) does not account for 
  residual energy in CH election.
  - PROBLEM: A node with 1% energy has the same chance of becoming a CH as 
    one with 100%.
  - IMPROVEMENT: Multiply T(n) by (E_residual / E_initial) to ensure only 
    energetic nodes take the high-load CH role.

* Member-to-CH Association:
  In 'processAdvMsg', nodes currently join based on RSSI. 
  - IMPROVEMENT: In UAV-assisted networks, nodes should ideally prioritize 
    CHs that are closer to the UAV's projected path (Trajectory-Aware Joining).

2. DATA FLOW BOTTLE-NECKS (SN -> CH -> UAV -> BS)
--------------------------------------------------------------------------------
* Aggregation Timing:
  CHs aggregate data into 'aggregatedDataSize'. 
  - PROBLEM: There is no clear "Buffer Overflow" logic. If the UAV is delayed 
    (e.g., in a large raster scan), CHs might deplete energy maintaining 
    large buffers or simply drop new packets.
  - FIX: Implement a FIFO queue in 'SensorNode' and define a 'MAX_BUFFER_SIZE'.

* UAV Discovery & Handshake:
  The UAV uses 'assessLink' and 'handshakeDelay'. 
  - PROBLEM: If 'handshakeDelay' + 'Treq' (transfer time) > 'Tcont' (contact 
    time), the transfer fails. 
  - IMPROVEMENT: Implement "Early Triggering." The CH should begin 
    preparing the data packet as soon as the UAV beacon is detected, 
    rather than waiting for a full handshake.

* BS Data Ingestion:
  'BaseStation.cc' parses packet IDs using string manipulation (std::stoi). 
  - PROBLEM: High computational overhead. For a network of 1000 nodes, 
    parsing long CSV strings of IDs at the BS will significantly slow down 
    simulation time.
  - FIX: Use 'omnetpp::cArray' or a 'std::vector<int>' directly as a 
    message parameter if the simulation allows, rather than string serialization.

3. METRIC RECORDING & ENERGY MODELING
--------------------------------------------------------------------------------
* UAV Energy Neglect:
  The 'MetricsCollector' tracks SN energy but flight energy for the UAV 
  is not integrated into total network energy.
  - IMPROVEMENT: Add a power model for the UAV: 
    P_flight = P_blade + P_profile + P_induced (dependent on 'searchSpeed').

* Stability Period (FND) Precision:
  'MetricsCollector.cc' records FND. 
  - PROBLEM: In a UAV-WSN, the "First Node Die" usually occurs in CHs 
    near the start of the UAV's path due to higher frequency of discovery.
  - IMPROVEMENT: Add "Spatial Death Distribution" metrics to see if 
    clustering is failing in specific geographic sectors.

* Throughput Calculation:
  Throughput is calculated at the BS. 
  - PROBLEM: This doesn't distinguish between "Data Sensed" vs "Data Collected". 
  - IMPROVEMENT: Track 'Buffer Occupancy' across all CHs to identify 
    where data is "stuck" in the network.

4. BREADCRUMB ROUTING (ALGORITHM 4) CRITIQUE
--------------------------------------------------------------------------------
* Breadcrumb Stale Data:
  The UAV collects 'L_nbr' (neighbor CH list). 
  - PROBLEM: If the UAV follows a breadcrumb to a neighbor CH, that neighbor 
    might have already been visited or its energy might have depleted.
  - FIX: Add a 'timestamp' or 'residualEnergy' to the breadcrumb info 
    so the UAV prioritizes the most "urgent" neighbors.

5. RECOMMENDED CODE REFACTORING
--------------------------------------------------------------------------------
1. Implement a 'GlobalCoordinator' module to handle synchronization of 
   LEACH rounds.
2. In 'UAVNode.cc', replace the constant 'flyTo' with a dynamic speed 
   controller (MaxSpeed for home-bound, SearchSpeed for raster).
3. Add a 'PacketDrop' counter in 'MetricsCollector' specifically for 
   "Contact Time Exceeded" events to measure UAV-Link reliability.
================================================================================

Specific Analysis of the Breadcrumb Mechanism
I noticed in UAVNode.cc and SensorNode.cc that you are using a "Breadcrumb" approach (Phase 3). This is an excellent addition to LEACH, as standard LEACH has no way of helping the Sink (UAV) find nodes. However, ensure that the CHs broadcast their beacons only to other CHs and not all Member Nodes; otherwise, the energy consumption for "Phase 3: Neighbor Discovery" will exceed the energy saved by the clustering itself.
