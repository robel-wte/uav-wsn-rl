1. THE "ZERO-MEMBER" CH PROBLEM: CAUSE & ANALYSIS
--------------------------------------------------------------------------------
The primary reason you see many CHs with 0 members is a RACE CONDITION in the 
OMNeT++ event scheduler between Phase 1 (Election) and Phase 2 (Joining).

* ISSUE A: Overlapping ADV and JOIN windows.
  In SensorNode.cc, if a node broadcasts an ADV_MSG, member nodes must wait 
  long enough to collect ALL ADV_MSGs before deciding on the 'max(RSSI)' CH. 
  If the 'startJoining' timer fires too early, nodes join the first CH they 
  hear, leaving later CHs with 0 members.

* ISSUE B: Lack of "CH-Acceptance" confirmation.
  The current logic assumes that if a CH is elected, it is "active." In reality,
  if a CH receives 0 JOIN_REQs, it should ideally step down or enter a low-power
  state to save the 0.5J budget.

2. LOGICAL FLAWS IN SensorNode.cc
--------------------------------------------------------------------------------
* FLAW 1: Static vs. Dynamic Thresholding
  The 'calculateThreshold()' function in your current code is likely missing
  the residual energy component. In a 400-node setup, high-density areas will
  produce too many CHs simultaneously, causing collisions of ADV_MSGs.
  FIX: Implement: T(n) = (P / (1 - P*(r mod 1/P))) * (E_residual / E_initial).

* FLAW 2: TDMA Schedule Prematurity
  CHs are likely creating their TDMA schedules (createTDMASchedule) before 
  the joining window has actually closed. This results in empty schedules.

3. LOGICAL FLAWS IN UAVNode.cc (SCALABILITY)
--------------------------------------------------------------------------------
* FLAW 3: Discovery Beacon Frequency
  With 400 nodes, if the UAV discovery beacon is too frequent, it saturates 
  the channel. CHs trying to respond to the UAV will collide with Member nodes
  trying to send data to CHs.

* FLAW 4: The "Blind" Raster Scan
  If a CH has 0 members, it still responds to the UAV. The UAV then wastes 
  time and energy (Algorithm 3) assessing a link for a node that has no 
  meaningful data to upload.

4. SUGGESTED ARCHITECTURAL FIXES
--------------------------------------------------------------------------------
[FIX 1] The "Minimum Member" Rule (SensorNode.cc):
Modify handleMessage so that if a CH has 0 members at the end of the 
joining phase, it resets its status to MEMBER_NODE for that round. 
This prevents the UAV from visiting "Ghost Clusters."

[FIX 2] Explicit Phase Timers:
Increase 'clusteringPhaseDuration' in omnetpp.ini to at least 5.0s. 
400 nodes need more "air time" to exchange ADV and JOIN packets without 
collisions (Carrier Sense Multiple Access overhead).

[FIX 3] RSI/RSSI Validation:
In 'processAdvMsg', add a distance check. If the CH is further than 
commRadius (100m), the node should ignore the ADV_MSG even if it is 
the only one it hears. This prevents "Orphan CHs" from being created 
by nodes that are too far away to actually connect.

[FIX 4] Metrics Precision:
In MetricsCollector.cc, the 'unclusteredNodes' calculation is currently 
averaging. It needs to be an absolute count per round to catch the 
"0-member" trend accurately in your CSV outputs.
